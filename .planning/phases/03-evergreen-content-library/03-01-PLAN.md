# Phase 03 Plan 01: Establish Content Platform Foundation

## Objective

Create a durable, migration-managed Neon content layer for topics, studies, versioned PDF documents, videos, canonical verses, topic membership, study/video relations, detected reference candidates, and editor-confirmed Scripture ranges.

## Context

- @.planning/PROJECT.md
- @.planning/REQUIREMENTS.md
- @.planning/ROADMAP.md
- @.planning/STATE.md
- @.planning/phases/03-evergreen-content-library/03-CONTEXT.md
- @.planning/research/2026-07-22-evergreen-rebuild-and-bible-v2-feasibility.md

## Tasks

1. Repair the planning workflow’s Windows PowerShell UTF-8 reads before it next rewrites `STATE.md`; the current script can display or propagate mojibake even when files are valid UTF-8.
2. Add an explicit Postgres migration workflow compatible with Neon; do not continue creating core content tables opportunistically during page requests.
3. Define typed tables for `topics`, `studies`, `study_documents`, `videos`, `canonical_books`, `canonical_chapters`, `canonical_verses`, `study_topics`, `video_topics`, `study_videos`, `study_reference_candidates`, and `study_scripture_references` with deliberate keys, uniqueness, indexes, and delete restrictions.
4. Model immutable PDF revisions with object-storage key, sanitized original filename, MIME type, byte size, SHA-256 checksum, extraction state/error, and a study-level pointer to the currently published document.
5. Model candidate and confirmed reference ranges separately. Confirmed rows must point to canonical start/end verses and support efficient “ranges containing this verse” queries.
6. Add shared validation for Hungarian titles/slugs, descriptions, PDF metadata, YouTube IDs/URLs, SEO fields, publication transitions, and Scripture ranges.
7. Build server-only repository functions for public published queries and separate authenticated admin queries. Public functions must make draft document/reference leakage structurally difficult.
8. Add idempotent database setup/seed entry points and document database plus object-storage environment variables and local/production migration commands.
9. Add small fixtures or database-independent tests for slug normalization, YouTube parsing, publication filtering, reference validation, range overlap, and document finalization invariants.

## Verification

- Run the migration against a disposable/local Postgres or Neon branch and inspect all keys, constraints, and indexes.
- Confirm the migration can run twice without corrupting data.
- Confirm public repository functions cannot return drafts.
- `npm run lint`
- `npm run build`
- `./scripts/gtd.ps1 verify -Phase 03`

## Success Criteria

- Every V1 entity and relationship has a typed, durable database representation.
- PDF revisions and their candidate lists cannot silently replace the currently published revision.
- Confirmed verse ranges can be queried efficiently by a canonical verse identity.
- Schema changes are versioned and reproducible rather than hidden in runtime reads.
- The memory workflow preserves Hungarian UTF-8 text when reading and rewriting planning files.
- Public and admin query boundaries are explicit.
- Hungarian slugs, YouTube URLs, SEO fields, and Scripture references are validated before persistence.
- README, summary, state, and verification evidence describe the new foundation.
